# Turf booking site

Book a turf slot: pick a time, verify your email with a code, pay with
Razorpay, get an instant e-ticket with a QR code, delivered by email.
No accounts, no passwords, no stored personal data beyond
`email + slot + payment id`.

## 1. Install

```bash
npm install
```

## 2. Set up accounts

- **Database**: create a free Postgres instance on [Supabase](https://supabase.com)
  or [Neon](https://neon.tech). Copy the connection string.
- **Razorpay**: create an account at [razorpay.com](https://razorpay.com),
  grab your test API keys from Settings > API Keys.
- **Resend**: create an account at [resend.com](https://resend.com) for
  sending the verification code and ticket emails. Free tier covers
  3,000 emails/month. Verify your own sending domain before going live —
  the shared `onboarding@resend.dev` address works for testing but is more
  likely to land in spam.

## 3. Configure environment

```bash
cp .env.example .env.local
```

Fill in `DATABASE_URL`, `RAZORPAY_KEY_ID`, `RAZORPAY_KEY_SECRET`,
`NEXT_PUBLIC_RAZORPAY_KEY_ID`, `RAZORPAY_WEBHOOK_SECRET`, `RESEND_API_KEY`,
`RESEND_FROM_EMAIL`.

## 4. Create the database table

```bash
psql "$DATABASE_URL" -f db/schema.sql
```

(Or paste `db/schema.sql` into the Supabase SQL editor.)

## 5. Run locally

```bash
npm run dev
```

Visit http://localhost:3000

## 6. Set up the Razorpay webhook (recommended)

In Razorpay dashboard > Settings > Webhooks, add:
`https://yourdomain.com/api/webhook/razorpay`, subscribed to
`payment.captured`. This confirms bookings even if a customer closes
the tab right after paying, before the page can call the verify API.

## 7. Deploy

Push to GitHub, import into [Vercel](https://vercel.com), add the same
environment variables in the Vercel dashboard, deploy.

## How data is handled

- No customer accounts, names, or addresses are ever stored.
- A booking row is only: `email, slot_date, slot_time, razorpay_payment_id, status`.
- Razorpay handles all card/UPI data directly — it never touches your server.
- Verification codes are generated by your own server (`lib/otpProvider.ts`),
  stored briefly in `email_otps` with a 10-minute expiry, and deleted once
  used — Resend only delivers the email, nothing more.
- Optional: uncomment the cleanup query at the bottom of `db/schema.sql` to
  auto-delete bookings after the slot date has passed.

## Things to review before going live

- Switch Razorpay from test to live keys.
- Adjust `ALL_SLOTS` in `app/api/slots/available/route.ts` to your real hours.
- Adjust `SLOT_PRICE_PAISE` in `.env.local` to your real price (or make it
  vary by day/time if needed — currently it's a flat rate).
- Consider WhatsApp instead of SMS for the ticket message (`lib/sendMessage.ts`)
  since it can carry the QR image directly.

## Homepage content to fill in (`app/page.tsx`)

The homepage ships with placeholder text and photos — replace before launch:

- **Photos**: add `turf-1.jpg` ... `turf-6.jpg` and `owner.jpg` to `public/images/`
  (see the README inside that folder).
- **Owner details**: name, bio, and direct contact in the "Who runs this place" section.
- **Rules**: the `rules` array at the top of `app/page.tsx` — edit freely to match
  your actual code of conduct.
- **Contact & grievance info**: phone, email, and address in the contact section.
- **Map location**: the embedded Google Map in the contact section uses a
  placeholder search query (`"Jammu, Jammu and Kashmir"`). Replace it in
  `app/page.tsx` — search for `Jammu, Jammu and Kashmir` (it appears twice,
  once for the embed and once for "Get directions") and swap in your real
  address or exact coordinates for a more precise pin. No Google API key
  is needed for this embed.

## Pre-payment terms dialog

Before Razorpay checkout opens, `components/PolicyModal.tsx` shows a dialog stating
the booking is non-refundable and that indiscipline results in removal and
forfeiture of the booking. The customer must click "I agree, continue to pay"
before checkout opens — declining cancels the flow with no charge. Edit the
wording in that file to match your actual policy.

## Owner dashboard

Visit `/admin` (also linked at the bottom of every page as "Owner login") to:

- **See confirmed bookings**: date, slot times, email address, payment ID (or
  "cash / walk-in" for manually added ones), and confirmation time. Filter by date.
- **Add a booking**: for walk-ins or bookings taken in person — pick a
  date, tap the open slots, optionally add an email, and it's marked
  confirmed immediately with no online payment involved.
- **Cancel a booking**: frees up the slot(s) immediately so they can be
  rebooked, online or in person.

Protected by a single password set as `ADMIN_PASSWORD` in `.env.local` —
pick something strong, since anyone with this password can see every
customer's email address and booking history, and can create or cancel
bookings. There's no per-staff login; if you need that later (e.g. separate
accounts for different staff), say so and it can be added.

## Discount codes

On `/admin`, the "Discount codes" panel lets you create codes like `SAVE10`
with a whole-number percentage off. Customers enter the code on the payment
page and click "Apply" before paying — the discount is always recalculated
server-side from the database at checkout, so nothing about the price can be
tampered with from the browser. Toggle a code off to stop it working without
losing its history; toggle it back on any time.

## Dev-only shortcuts — remove before deploying

Two buttons let you test the booking flow locally without needing Resend or
Razorpay set up:

- **"Skip OTP"** on the verify page
- **"Skip payment"** on the payment page

Both are wrapped in clearly marked comments:
```
// ===== DEV-ONLY BLOCK — safe to delete before deploying =====
...
// ===== END DEV-ONLY BLOCK =====
```
in `app/verify/page.tsx` and `app/payment/page.tsx` — search for that marker
in each file and delete everything between the two lines (the function and
the button) before you go live.

They're already automatically hidden in a production build (`NODE_ENV !==
"production"` gates both), and the API route they call
(`app/api/dev/confirm/route.ts`) refuses to run in production even if
someone finds the URL directly — so nothing breaks if you forget to remove
them. But for a clean production codebase, also delete the whole
`app/api/dev/` folder once you're ready to deploy for real.

## Per-slot pricing & adding new time slots

Prices are no longer one flat rate — each time slot has its own price,
managed in `/admin` under "Time slots & pricing":

- **Add a new slot**: pick a time and price, click "Add / update slot."
- **Change a price**: enter the same time again with a new price — it
  updates rather than duplicating.
- **Disable/Enable**: temporarily stop offering a slot without losing its
  history.
- **Delete**: removes it from the bookable list entirely. This never
  affects bookings already made — each booking stores the exact price it
  was charged at the time, so changing or removing a slot later never
  retroactively changes what a past customer paid.

## Revenue

Also on `/admin`, the "Revenue" panel shows real money received through
Razorpay for a date range — click "Today" for a quick daily total, or set
a custom "From"/"To" range (e.g. April 2 to April 10) and click
"Calculate." Only genuine online payments count; walk-in/cash bookings
added from the admin panel are excluded, since that money never passed
through the site.
